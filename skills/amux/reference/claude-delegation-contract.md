# Experimental read-only Claude delegation contract

This is the single normative contract for the skill-owned local experiment. It has **no compatibility guarantee**, adds no `amux` CLI resource, and Claude never becomes an Amp worker, runner, group member, or generic CLI resource. It is not a synthetic thread. Use it only after an explicit request to delegate read-only analysis. The helper is [`../experimental/claude-delegation/claude_delegation.py`](../experimental/claude-delegation/claude_delegation.py).

## Authority and launch surface

A **thinker** is one fresh, delegation-scoped, full interactive Claude Code process in a dedicated same-machine tmux window. Its authority is `read_only`: it may use only Claude's `Read`, `Grep`, and `Glob` tools plus the experiment's bounded semantic-submission MCP tools. The launch denies shell, edit, write, notebook, agent, web, skill, non-managed inherited settings and hooks, extra directories, slash commands, and automatic interactive input. Managed policy has higher precedence and cannot be disabled by session settings, so its runtime effect remains explicit diagnostic evidence rather than an assumed guarantee. This is policy confinement, not an OS sandbox; disclose both limitations and constrain the supplied sources accordingly.

The coordinator supplies a private UTF-8 launch packet. Never commit or copy that packet, prompts, transcripts, pane captures, tool streams, secrets, raw receipts, or complete artifacts. Pane text is diagnostic only. A valid helper-submitted semantic message is authoritative.

Before a read-only packet is finalized, `launch policy-digest` returns the deterministic digest of the read-only launch policy without reading a packet, inspecting a worktree or tmux, or writing private state. It rejects other workflows. A read-only request may either omit model selection, preserving the existing default-model argv and policy digest exactly, or carry one exact owner-authorized approved identifier: `claude-fable-5` or `claude-opus-4-8`; the helper does not normalize, alias, choose, fall back, or retry models. An explicit model is included in the policy digest, exact Claude argv, normalized argv digest, launch-command transport digest, launch request digest, launch intent, and immutable receipt binding. The final packet must include that exact digest in every semantic envelope template, and the final read-only launch request must carry the same value as `expected_launch_policy_digest`. `launch plan` validates that expectation and any model selection against the selected policy before any platform, tmux, packet, or worktree probe, then verifies a supported exact-process-identity implementation, tmux, the installed Claude flags, exact canonical repository, base commit, clean dedicated linked worktree without optional Git locks, and deterministic packet, launch-policy, and launch-command digests. Packet and transport reads open one non-followed, nonblocking descriptor, require one regular file owned by the effective user with exact mode `0600`, inspect it with `fstat`, and perform one bounded read; packet device/inode identity is transitively bound by the launch-command digest, rechecked under the final lock, retained through startup, and checked again by transport through its private source route. Worktree validation is performed while `fchdir` is anchored to one non-followed directory descriptor; that final descriptor is retained through startup, and its device/inode identity is transported and rechecked before transport-side `fchdir`. Every accepted packet uses one bounded owner-only canonical runtime transport containing the exact Claude argv, canonical workdir and object identity, verified launcher device/inode/content identity, and a captured execution environment after the workflow's credential-removal policy. The launch environment is derived before every launch `--version` and `--help` probe and is reused for the final process, so mutating credential removal covers all three; diagnostic probes always use the credential-removed environment while preserving benign entries. Its SHA-256 digest is carried by and verified against the bounded launch command, so the immutable launch-command digest transitively binds that complete execution tuple. Preflight checks each process string where the platform exposes a separate limit and conservatively accounts for all argv/environment bytes, terminating NULs, pointers, and a fixed margin against `ARG_MAX`; the transport repeats those checks before execution. The global and target-session tmux environment snapshots are also conservatively budgeted for the helper process (including both snapshots rather than assuming overrides), their digest is bound into the helper command, and execute rechecks that exact digest under the final lock. A packet, environment, object, command, or model that cannot satisfy those checks is rejected during plan/execute preflight before receipt, runtime, launch-intent, tmux, or process mutation. Darwin derives the verified source descriptor's complete content identity, makes a private probe copy, reopens the exact destination without following links, and requires matching stable size/content identity before sealing and pathname execution; after intent, the parent makes another copy at the transport-bound route inside only that delegation's private runtime container, opens exact retained descriptors for the destination and container, sets both objects user-immutable while making the container non-writable, and verifies the copied destination's stable complete content before starting tmux. Transport reopens that exact destination without following links, verifies stable content and both immutable flags, and retains the destination descriptor through pathname execution. The parent clears the immutable flags and restores only that original descriptor-bound delegation container after every tmux/startup outcome; shared state and runtime namespaces remain writable for independent delegations. Linux executes the verified executable descriptor and keeps a node/bun script descriptor available for live identity checks; startup, acquisition, revalidation, and parking open the exact kernel `/proc/<pid>/fd/<n>` route without resolving it back to a pathname, stably hash the live object, and require its complete device/inode/size/content identity. Darwin uses its existing kernel process APIs. Linux read-only launch additionally requires stable `/proc/<pid>/stat` start time, NUL-delimited `/proc/<pid>/cmdline`, and `/proc/<pid>/exe` target/device/inode identity; those sources are read twice and must agree. Missing, denied, changed, recycled, zombie, truncated, or ambiguous data fails closed. Linux mutating launch remains unavailable. The plan is a dry run: it writes no receipt, runtime configuration, tmux window, or Git index metadata; private temporary probe copies are sealed while executing and removed before it returns. Missing or mismatched expectation stops the workflow before receipt creation. `launch execute` validates model syntax and approval before receipt inspection, then rejects a presence or value mismatch against the immutable receipt before platform/session probes or any receipt/runtime mutation. It next requires the immutable digests and stable event ID to match. Under the final receipt lock it revalidates the target session, packet bytes and descriptor identity, executable object/content identity, both execution budgets, and the descriptor-anchored read-only worktree or mutating lease before reserving launch intent; those packet/worktree/executable descriptors remain open until verified startup or process exit. It writes owner-only transport/MCP/settings files, releases the receipt lock so transport can take the final lifecycle-to-receipt authorization boundary, and creates one detached tmux window. After startup verification it reacquires the receipt lock and records `launch_completed` only if the receipt remains the exact reserved launch-indeterminate state and that exact pane was repeatedly observed for at least 1.5 seconds running the expected Claude session with the bound normalized argv, executable content/policy, workdir, and launch command. Startup failure, substitution, concurrent receipt mutation, or timeout leaves only `launch_intent` and is genuinely indeterminate; it never records false completion or authorizes relaunch. A completed exact replay returns the original tmux identity without relaunching; an intent lacking a result is indeterminate and forbids automatic relaunch.

Private tmux metadata may contain only the canonical workdir, helper/private-state route, opaque delegation ID, and bounded correlation digests needed to start the helper. It must not contain packet text or the packet source path. The owner-only transport may contain the packet source route needed for post-intent object revalidation. Public helper errors describe the failed invariant without including private packet, transport, or runtime paths.

## Receipt and event rules

All receipt mutations use one blocking `experimental.lock` domain. Receipt creation also uses the canonical lifecycle registry's lock: it registers every private state store before committing a receipt and rejects an origin with a durable teardown fence. The canonical registry is the authoritative discovery seam even when `--state-dir` selects an alternate private receipt store; isolated test state is unavailable to normal helper invocations. The private state parent is mode `0700`; lock, lifecycle registry, receipt, packet transport, and generated data files are mode `0600`, while a verified private executable copy is mode `0500`. Commits flush a temporary file, atomically replace the receipt file, and flush its parent directory.

The immutable binding contains protocol version, delegation ID, nonce, task/question identities, opaque origin Amp thread, repository/base/canonical workdir, thinker/read-only authority, bounded task reference, packet/policy/launch digests, and the explicit read-only model when selected. The origin thread is the exact immutable origin-thread binding to one Amp worker identity; names, cwd, PID, issue number, tmux session, window, and Claude session ID never establish or repair ownership. Omitted model bindings remain byte-compatible with the existing default behavior. Mutable routing contains only current target and optional machine-local recovery route. Routing changes never rewrite provenance or binding. Mutating delegation bindings and requests do not accept model selection.

Every event has a caller-supplied stable ID. Exact replay returns `duplicate`; reuse for different content fails before mutation. Events remain append-only even though the receipt also materializes current state. Report transitions are exactly:

`valid_report → delivered → acknowledged → verified_parked`

The machine-local inbox consumption creates delivery. Coordinator acknowledgement is separate and only permits parking. A wake-up notification is not delivery or acknowledgement.

An acknowledged and identity-verified parked receipt records a date 30 days later when it becomes cleanup-eligible. Paired worker teardown preserves that 30-day cleanup eligibility, the receipt and all event/report/group history; eligibility never authorizes deletion of private artifacts or worktrees.

## Semantic envelopes

Both `thinker_report` and `input_request` envelopes bind protocol version, delegation ID, nonce, message ID, `in_reply_to`, task ID, origin thread, repository, base, workdir, thinker/read-only authority, launch-policy digest, and timestamp to the receipt.

A report explicitly accepts the role and exclusions and records `complete` or `blocked`, bounded verdict and rationale, and bounded lists of evidence, assumptions, unsupported claims, blockers, verification, and references. `changed_artifacts` must be empty. Validation establishes shape and correlation, not truth or acceptance.

An `input_request` records one of `clarification`, `decision`, or `missing_evidence`, plus a bounded question and blocking reason. Inbox consumption may mark it seen but not resolved. Only an explicitly observed correlated Claude-side acceptance (`input accept`) or a later explicit superseding source event resolves it. There is no automatic response injection or automatic follow-up.

Semantic fields must contain concise non-content evidence only. Keys for prompts, transcripts, pane captures, tool streams, secrets, artifact content, and complete artifacts are rejected recursively. This enforces shape, not meaning: forbidden content embedded in an otherwise allowed string cannot be detected mechanically and still requires deliberate privacy review. References must point only to deliberately allowed sources; never use private operational evidence as a reference.

## Identity, notification, and diagnostics

Session acquisition requires the exact tmux session/window/window ID/pane ID and verified startup executable identity recorded by this receipt's single `launch_completed` event. It rechecks the intent-bound complete normalized Claude argv rather than only the session-ID pair, while accepting the supported direct Claude form, Linux node/bun Claude Code form, and the exact Darwin transport form whose process and kernel executable are the helper-created `verified-claude` private copy while absolute `argv[0]` names the planned launcher. That Darwin form requires both the planned launcher's complete device/inode/size/content identity and raw absolute path digest and the completed private copy's complete device/inode/size/content identity; either observed `verified-claude` marker requires this complete form and cannot fall through to direct or generic wrapper support. It then binds pane PID and creation identity, canonical workdir, platform kernel process start and executable identity, exact NUL-delimited live argv digest, one exact Claude `--session-id` pair, and launch-command digest. A policy flag, packet, executable, wrapper, pane, or process substitution is rejected even if it retains the session ID. Explicit parking is allowed only after acknowledgement with no unresolved input request. It durably reserves intent, re-verifies that complete identity immediately before killing the exact pane, and records a separate result. PID reuse, changed argv or executable, changed tmux identity, or unavailable identity data blocks the kill. An interrupted intent requires explicit `{"recover":true}` reconciliation; exact confirmed pane absence may complete that authorized intent, while ambiguous inspection remains unresolved. Failure preserves the acknowledged receipt for recovery.

`lifecycle worker-teardown` is a skill-owned paired preflight/cleanup operation, not an `amux` resource command. It selects receipts only by their exact immutable origin thread across every canonically registered store. Execution first persists an origin-scoped teardown fence under the lifecycle lock, so concurrent or later creation for that worker fails closed through the gap before Amp teardown. A registered store is strict durable evidence: a missing or unreadable registered directory or `receipts.json` blocks rather than degrading to an empty store; only the legitimate never-used canonical default may be absent. Dry-run, execution, and fence release return only the origin-thread SHA-256, SHA-256 pair handles, and non-content action and blocker codes; they never emit raw origin threads, stored delegation IDs, packet/report content, prompts, account identity, credentials, tokens, private paths, or raw argv. No pair and fully event-verified `verified_parked` pairs are safe. An acknowledged pair with no unresolved input or prior park intent, exactly correlated launch intent/completion/acquisition and report/delivery/acknowledgement events, materialized state equality, and full current identity equality may be parked under explicit worker-teardown authority. The helper preflights every bound pair before any park. Every active, unacknowledged, unresolved, mismatched, missing, malformed, launch-indeterminate, park-indeterminate, or otherwise unverified state blocks before Amp worker teardown and preserves all panes, artifacts, worktrees, receipts, reports, and group history. A cleanup-time failure records or preserves the durable park intent, returns a blocker, and requires the existing explicit recovery route. Exact replay reuses the fence and deterministic park event. The fence survives interruption and Amp teardown failure; explicit `worker-teardown-release` may remove it only while every pair still proves safely parked. Only a successful paired operation permits the separate Amp worker teardown, which remains the final action.

`lifecycle register-legacy-store --origin-thread <thread-id> --store-path <exact-private-store>` is the only legacy discovery seam. It accepts only an explicitly spelled absolute canonical path whose final directory, existing lock, and `receipts.json` are descriptor-opened relative to the retained directory without following links, owned by the effective user, and exactly mode `0700`/`0600`. Under canonical lifecycle-then-store lock order, the same bounded stable descriptor read must decode one supported receipt schema containing at least one receipt and no receipt with an immutable origin different from the supplied exact origin. Missing, unreadable, oversized, malformed, mixed-origin, symlinked, wrong-owner, or wrong-mode evidence blocks. Success additively records the exact store path and directory object identity in the canonical registry; exact replay is idempotent only for that same object and no receipt is rewritten. Later discovery blocks if the registered path denotes another object. Output contains only the origin and store-object hashes plus bounded codes. No name, cwd, PID, issue, tmux, session, glob, or repository inference is permitted.

`lifecycle detach-indeterminate-worker` is the explicit recovery seam for one registered receipt with exactly one `launch_intent` and no completed launch, session, message, input, report, acknowledgement, or park transition. Its JSON request supplies the receipt's exact delegation ID and immutable origin, one stable event ID, and `authorization` containing a bounded terminal state (`merged` or `closed_terminal`), a terminal report SHA-256, and coordinator-authorization SHA-256. Those digests are trusted evidence references supplied by the authorized operator; this local helper validates their exact SHA-256 shape and durably binds them to the operation, but does not retrieve the referenced report or independently establish coordinator authority. Before inspection it creates or retains the durable origin fence. While retaining the receipt lock, it attempts the delegation's durable launch gate only for a short bounded interval. A busy gate returns `launch_transport_active_or_indeterminate` and promptly releases receipt and lifecycle locks without receipt mutation; detach never waits indefinitely for a transport or descendant while holding those locks. Only after acquiring the gate does it perform bounded all-pane tmux and same-owner process inventories and retain the gate through the append-only detach commit. Both inventories read at most the configured byte cap plus one directly from a child pipe, discard stderr, enforce a timeout, and kill/wait on failure; malformed UTF-8, over-limit output, timeout, launch failure, or unavailable inspection blocks without exposing output. The launch transport holds the same gate and requires the receipt to remain in the exact pre-completion launch-indeterminate state with no durable origin fence. At the final pre-exec boundary it acquires canonical lifecycle then distinct receipt mutation locks, repeats authorization, and calls `execve` while those close-on-exec lock descriptors remain held; a fence therefore either commits before final authorization and rejects execution or can commit only after execution has begun. The gate is close-on-exec and is released by successful exec rather than inherited by the launched target; a failed or pre-commit exec retains it until transport unwinds. The temporary gate-to-lifecycle inversion cannot deadlock detach because detach's gate attempt is bounded and releases its lifecycle/receipt locks on a busy result. A retained transport therefore cannot execute after launch completion, parking, teardown fencing, or detach; an active or indeterminate pre-exec transport makes detach block rather than waiting, while a completed target does not keep the transport gate busy for its lifetime. Detach accepts absence only when the exact target is absent and no process carries the intent-bound Claude session and argv; a matching process remains a blocker even after a pane rename or move. A matching live process, any target or same-session process whose full launch identity mismatches or cannot be proven, malformed or over-limit inventory, or unavailable inspection blocks. Success appends one final `worker_detached` event and additive materialized proof while preserving the receipt state, immutable origin, launch intent, packet/runtime/artifacts/worktree, and fence. The detached receipt is sealed under its mutation lock: every launch, acquisition, routing, message, input, delivery, acknowledgement, notification, park, recovery, and failure mutation rejects before writing. Only read inspection, exact detach replay, and paired teardown remain available. The final-event validation remains a tamper defense. Detach creates no launch completion, session acquisition, report, delivery, acknowledgement, park, cleanup eligibility, process input, pane kill, retry, or adoption. Exact replay is duplicate-safe; conflicting event reuse and a second detach identity fail closed. JSON output exposes only hashes, fence status, and bounded outcome/absence/blocker codes.

The compatibility selector `"compatibility":"pre_identity_launch_intent_v1"` extends that explicit seam only for a registered receipt whose sole launch intent has the exact historical pre-identity shape: the event binds its workflow, request, Claude session, tmux session/window, packet, launch policy, launch command, and timestamp, contains none of the later argv/launcher/executable or packet/worktree object-identity fields, and exactly matches the receipt's immutable digests and producer role.
