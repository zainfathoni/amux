# Experimental read-only Claude delegation contract

This is the single normative contract for the skill-owned local experiment. It has **no compatibility guarantee**, adds no `amux` CLI resource, and does not make Claude an Amp worker, runner, group member, or synthetic thread. Use it only after an explicit request to delegate read-only analysis. The helper is [`../experimental/claude-delegation/claude_delegation.py`](../experimental/claude-delegation/claude_delegation.py).

## Authority and launch surface

A **thinker** is one fresh, delegation-scoped, full interactive Claude Code process in a dedicated same-machine tmux window. Its authority is `read_only`: it may use only Claude's `Read`, `Grep`, and `Glob` tools plus the experiment's bounded semantic-submission MCP tools. The launch denies shell, edit, write, notebook, agent, web, skill, non-managed inherited settings and hooks, extra directories, slash commands, and automatic interactive input. Managed policy has higher precedence and cannot be disabled by session settings, so its runtime effect remains explicit diagnostic evidence rather than an assumed guarantee. This is policy confinement, not an OS sandbox; disclose both limitations and constrain the supplied sources accordingly.

The coordinator supplies a private UTF-8 launch packet. Never commit or copy that packet, prompts, transcripts, pane captures, tool streams, secrets, raw receipts, or complete artifacts. Pane text is diagnostic only. A valid helper-submitted semantic message is authoritative.

Before a read-only packet is finalized, `launch policy-digest` returns the deterministic digest of the read-only launch policy without reading a packet, inspecting a worktree or tmux, or writing private state. It rejects other workflows. A read-only request may either omit model selection, preserving the existing default-model argv and policy digest exactly, or carry the one exact approved identifier `claude-fable-5`; the helper does not normalize, choose, fall back, or retry models. An explicit model is included in the policy digest, exact Claude argv, normalized argv digest, launch-command transport digest, launch request digest, launch intent, and immutable receipt binding. The final packet must include that exact digest in every semantic envelope template, and the final read-only launch request must carry the same value as `expected_launch_policy_digest`. `launch plan` validates that expectation and any model selection against the selected policy before any platform, tmux, packet, or worktree probe, then verifies a supported exact-process-identity implementation, tmux, the installed Claude flags, exact canonical repository, base commit, clean dedicated linked worktree without optional Git locks, and deterministic packet, launch-policy, and launch-command digests. Packet and transport reads open one non-followed, nonblocking descriptor, require one regular file owned by the effective user with exact mode `0600`, inspect it with `fstat`, and perform one bounded read; packet device/inode identity is transitively bound by the launch-command digest, rechecked under the final lock, retained through startup, and checked again by transport through its private source route. Worktree validation is performed while `fchdir` is anchored to one non-followed directory descriptor; that final descriptor is retained through startup, and its device/inode identity is transported and rechecked before transport-side `fchdir`. Every accepted packet uses one bounded owner-only canonical runtime transport containing the exact Claude argv, canonical workdir and object identity, verified launcher device/inode/content identity, and a captured execution environment after the workflow's credential-removal policy. The launch environment is derived before every launch `--version` and `--help` probe and is reused for the final process, so mutating credential removal covers all three; diagnostic probes always use the credential-removed environment while preserving benign entries. Its SHA-256 digest is carried by and verified against the bounded launch command, so the immutable launch-command digest transitively binds that complete execution tuple. Preflight checks each process string where the platform exposes a separate limit and conservatively accounts for all argv/environment bytes, terminating NULs, pointers, and a fixed margin against `ARG_MAX`; the transport repeats those checks before execution. The global and target-session tmux environment snapshots are also conservatively budgeted for the helper process (including both snapshots rather than assuming overrides), their digest is bound into the helper command, and execute rechecks that exact digest under the final lock. A packet, environment, object, command, or model that cannot satisfy those checks is rejected during plan/execute preflight before receipt, runtime, launch-intent, tmux, or process mutation. Darwin derives the verified source descriptor's complete content identity, makes a private probe copy, reopens the exact destination without following links, and requires matching stable size/content identity before sealing and pathname execution; after intent, the parent makes another copy at the transport-bound route inside only that delegation's private runtime container, opens exact retained descriptors for the destination and container, sets both objects user-immutable while making the container non-writable, and verifies the copied destination's stable complete content before starting tmux. Transport reopens that exact destination without following links, verifies stable content and both immutable flags, and retains the destination descriptor through pathname execution. The parent clears the immutable flags and restores only that original descriptor-bound delegation container after every tmux/startup outcome; shared state and runtime namespaces remain writable for independent delegations. Linux executes the verified executable descriptor and keeps a node/bun script descriptor available for live identity checks; startup, acquisition, revalidation, and parking open the exact kernel `/proc/<pid>/fd/<n>` route without resolving it back to a pathname, stably hash the live object, and require its complete device/inode/size/content identity. Darwin uses its existing kernel process APIs. Linux read-only launch additionally requires stable `/proc/<pid>/stat` start time, NUL-delimited `/proc/<pid>/cmdline`, and `/proc/<pid>/exe` target/device/inode identity; those sources are read twice and must agree. Missing, denied, changed, recycled, zombie, truncated, or ambiguous data fails closed. Linux mutating launch remains unavailable. The plan is a dry run: it writes no receipt, runtime configuration, tmux window, or Git index metadata; private temporary probe copies are sealed while executing and removed before it returns. Missing or mismatched expectation stops the workflow before receipt creation. `launch execute` validates model syntax and approval before receipt inspection, then rejects a presence or value mismatch against the immutable receipt before platform/session probes or any receipt/runtime mutation. It next requires the immutable digests and stable event ID to match. Under the final receipt lock it revalidates the target session, packet bytes and descriptor identity, executable object/content identity, both execution budgets, and the descriptor-anchored read-only worktree or mutating lease before reserving launch intent; those packet/worktree/executable descriptors remain open until verified startup or process exit. It then writes owner-only transport/MCP/settings files and creates one detached tmux window. `launch_completed` is recorded only after that exact pane is repeatedly observed for at least 1.5 seconds running the expected Claude session with the bound normalized argv, executable content/policy, workdir, and launch command. Startup failure, substitution, or timeout leaves only `launch_intent` and is genuinely indeterminate; it never records false completion or authorizes relaunch. A completed exact replay returns the original tmux identity without relaunching; an intent lacking a result is indeterminate and forbids automatic relaunch.

Private tmux metadata may contain only the canonical workdir, helper/private-state route, opaque delegation ID, and bounded correlation digests needed to start the helper. It must not contain packet text or the packet source path. The owner-only transport may contain the packet source route needed for post-intent object revalidation. Public helper errors describe the failed invariant without including private packet, transport, or runtime paths.

## Receipt and event rules

All receipt mutations use one blocking `experimental.lock` domain. The private state parent is mode `0700`; lock, receipt, packet transport, and generated data files are mode `0600`, while a verified private executable copy is mode `0500`. Commits flush a temporary file, atomically replace the receipt file, and flush its parent directory.

The immutable binding contains protocol version, delegation ID, nonce, task/question identities, opaque origin Amp thread, repository/base/canonical workdir, thinker/read-only authority, bounded task reference, packet/policy/launch digests, and the explicit read-only model when selected. Omitted model bindings remain byte-compatible with the existing default behavior. Mutable routing contains only current target and optional machine-local recovery route. Routing changes never rewrite provenance or binding. Mutating delegation bindings and requests do not accept model selection.

Every event has a caller-supplied stable ID. Exact replay returns `duplicate`; reuse for different content fails before mutation. Events remain append-only even though the receipt also materializes current state. Report transitions are exactly:

`valid_report → delivered → acknowledged → verified_parked`

The machine-local inbox consumption creates delivery. Coordinator acknowledgement is separate and only permits parking. A wake-up notification is not delivery or acknowledgement.

An acknowledged and identity-verified parked receipt records a date 30 days later when it becomes cleanup-eligible. Eligibility never authorizes deletion; this experiment performs no automatic cleanup.

## Semantic envelopes

Both `thinker_report` and `input_request` envelopes bind protocol version, delegation ID, nonce, message ID, `in_reply_to`, task ID, origin thread, repository, base, workdir, thinker/read-only authority, launch-policy digest, and timestamp to the receipt.

A report explicitly accepts the role and exclusions and records `complete` or `blocked`, bounded verdict and rationale, and bounded lists of evidence, assumptions, unsupported claims, blockers, verification, and references. `changed_artifacts` must be empty. Validation establishes shape and correlation, not truth or acceptance.

An `input_request` records one of `clarification`, `decision`, or `missing_evidence`, plus a bounded question and blocking reason. Inbox consumption may mark it seen but not resolved. Only an explicitly observed correlated Claude-side acceptance (`input accept`) or a later explicit superseding source event resolves it. There is no automatic response injection or automatic follow-up.

Semantic fields must contain concise non-content evidence only. Keys for prompts, transcripts, pane captures, tool streams, secrets, artifact content, and complete artifacts are rejected recursively. This enforces shape, not meaning: forbidden content embedded in an otherwise allowed string cannot be detected mechanically and still requires deliberate privacy review. References must point only to deliberately allowed sources; never use private operational evidence as a reference.

## Identity, notification, and diagnostics

Session acquisition requires the exact tmux session/window/window ID/pane ID and verified startup executable identity recorded by this receipt's single `launch_completed` event. It rechecks the intent-bound complete normalized Claude argv rather than only the session-ID pair, while accepting the supported direct Claude form, Linux node/bun Claude Code form, and the exact Darwin transport form whose process and kernel executable are the helper-created `verified-claude` private copy while absolute `argv[0]` names the planned launcher. That Darwin form requires both the planned launcher's complete device/inode/size/content identity and the completed private copy's complete device/inode/size/content identity; it is not generic wrapper support. It then binds pane PID and creation identity, canonical workdir, platform kernel process start and executable identity, exact NUL-delimited live argv digest, one exact Claude `--session-id` pair, and launch-command digest. A policy flag, packet, executable, wrapper, pane, or process substitution is rejected even if it retains the session ID. Explicit parking is allowed only after acknowledgement with no unresolved input request. It durably reserves intent, re-verifies that complete identity immediately before killing the exact pane, and records a separate result. PID reuse, changed argv or executable, changed tmux identity, or unavailable identity data blocks the kill. An interrupted intent requires explicit `{"recover":true}` reconciliation; exact confirmed pane absence may complete that authorized intent, while ambiguous inspection remains unresolved. Failure preserves the acknowledged receipt for recovery.

Optional Amp wake-up first requires `amp inspect` of one exact live pane whose tmux and process arguments identify the expected origin thread. Notification sends only SHA-256 correlation tokens after the report commit. Its durable intent prevents concurrent or interrupted replay from sending another token. If an interrupted attempt has no result, exact replay durably records deterministic `unavailable` evidence without resending. Missing, stale, ambiguous, or changed identity fails closed and leaves the machine-local inbox recoverable. Notification is not delivery.

`diagnose` is read-only and classifies each capability as `supported`, `unavailable`, or `untested`. Its model-selection result proves only that the installed Claude CLI exposes `--model`; it is not evidence that Fable 5 or any other model is provisioned, available, affordable, selected autonomously, or capacity-safe. The documented CodexBar usage payload is unversioned, so it cannot establish supported capacity provenance: diagnostics validate its bounded public-safe shape, then report capacity unavailable without projecting values or inventing source/schema versions. A future source becomes supported only with a genuine exact provider, source-contract version, schema discriminator, and fresh bounded utilization windows. Account and subscription identity is excluded. Unknown, stale, or structurally mismatched capacity is not autonomous evidence. An untested runtime capability must never be promoted to supported by assumption.
