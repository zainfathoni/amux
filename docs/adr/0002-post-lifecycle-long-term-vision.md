# Post-lifecycle long-term vision and horizons

amux's long-term direction after the agent-first client lifecycle work is recorded here as proposed direction, not accepted work. ADR 0001 remains the only accepted plan; the in-flight lifecycle delivery is unchanged by this document, and on any conflict ADR 0001 and its contracts take precedence. Each horizon below becomes real only through its own future accepted ADR — never by editing ADR 0001 or re-scoping in-flight work.

This vision was distilled from a dedicated strategy session and selectively draws from [Tycho](https://github.com/firewalker06/tycho)'s visibility and [Gas City](https://gascity.com/)'s agent-coordination ideas. Neither project is a target architecture: amux keeps only the pieces that fit its interactive, tmux-native, personal-first workflow and rejects Gas City's software-factory complexity, resident supervision, exhaustive orchestration state, and associated token overhead.

## Identity and durable bets

- amux is a personal-first tool, polished. Felt friction in the maintainer's own workflows drives priorities, while core contracts remain structurally reusable when doing so does not compromise those workflows. Other users may adopt or copy it, but amux does not absorb deployment topology: machine-specific topology and orchestration belong in the maintainer's private `fleet` repository.
- amux is Amp-first with a deliberately asymmetric Claude Code seam. A future parent concept may group interactive clients for aggregate lifecycle and attention behavior, but ADR 0001 workers remain specifically Amp TUI clients identified by canonical Amp thread ID; separately named Claude Code clients do not redefine that accepted resource.
- Claude Code support is scoped to creating, launching, parking, restarting, and removing interactive TUI clients in tmux windows. Shelve and teardown remote-state semantics stay Amp-only; degraded emulation of Amp archive state was explicitly rejected, as were Claude runner equivalents, scheduled Claude runs, and independent Claude orchestration.
- Amp remains the main driver and coordination surface. The economic purpose of Claude Code clients is to consume otherwise-idle subscription capacity on selected delegated tasks, then report back to the originating Amp thread. Backend parity and volume machinery are not goals.
- The Amp coordinator may autonomously select Claude for a deliberate single delegation, but never fans out or fills capacity merely because it exists. Selection is capacity-aware: on macOS it may read provider-reported five-hour and seven-day utilization, reset times, confidence, and pace through CodexBar's machine-readable CLI; an official Claude machine-readable surface takes precedence if one becomes available. Delegation targets capacity likely to expire unused while preserving separately configured hard floors for the five-hour, weekly, and applicable model-specific windows; the most constrained window governs. Missing or low-confidence capacity data disables autonomous delegation rather than risking that reserve, while an explicit user request may proceed after acknowledging the unknown capacity.
- The first delegation experiment is macOS-first because that is the maintainer's capacity-bearing workflow. Receipt and adapter contracts remain portable, but Linux exposes only capabilities its local Claude installation and usage sources can prove; reduced lifecycle, capacity, or delivery support is preferable to simulated parity.
- Horizon-one delegation stays on the originating Amp coordinator's machine. Runner availability never implies Claude dispatch eligibility, and automatic or explicit cross-machine selection remains part of the later remote-reach decision.
- Claude adoption begins as a staged, skill-owned experiment rather than a first-class lifecycle API: one local read-only delegation, one isolated local mutating delegation, optional teleport or adoption experiments, then a separate promotion decision. Each uses an explicit `/amux` workflow to launch a full interactive Claude client in a verified same-machine tmux window, record bounded capacity and outcome evidence, and retain manual recovery. The experiment compares its results with the fixed #105 and #118 contracts and reuses durable-write-before-notification and verified-lease invariants where they fit. The shipped report command remains Amp-work-group-specific, so the experiment neither changes it nor makes a Claude session an Amp work-group member.
- The first pilots form a staged pair drawn from real work rather than synthetic demos. A bounded read-only delegation first validates launch, capacity capture, question handling, and report-back without Git ownership as a confounder. Only after that loop succeeds does a mutating pilot use a dedicated worktree and require a clean-commit handoff, isolating the additional workdir and artifact-contract questions.
- Advancement to the mutating pilot requires one successful real read-only task against explicit criteria: capacity capture; continuously verifiable client identity; durable semantic report persistence; successful notification or manual recovery to the origin Amp thread; a result useful enough to consume; and acknowledgement-gated parking. Optional paths that did not naturally occur, such as an input request or failed notification, are recorded as untested rather than assumed reliable and do not require synthetic exercise before the next pilot.
- The pilot uses a tiny experimental receipt helper because #118's delivered `report submit` contract requires a canonical Amp member thread and work-group-specific `ready`, `blocked`, or `merged` semantics. Claude instead submits a structured report under the delegation correlation identity; the helper serializes mutations, validates idempotent event transitions, crash-durably persists bounded private state, and only then attempts an optional verified best-effort notification. Failed or unavailable notification leaves the report recoverable through `/amux`. Its storage shape is explicitly unstable and is evaluated against #118's transport invariants rather than forced through its workflow API; successful notification is never treated as acknowledgement.
- The helper, its unstable report schema, and the `/amux` orchestration that invokes it are versioned in a clearly marked experimental area alongside the amux skill so their evolution can inform later API design. They do not enter lifecycle CLI core and carry no compatibility guarantee. Raw receipts, capacity measurements, and task outcomes remain bounded private local data rather than repository artifacts.
- Claude authors semantic outcomes explicitly by invoking the helper with its structured report. A Claude Code hook validates that the correlated receipt exists, contains required fields, and satisfies objective invariants such as the declared handoff shape before the session can be considered completed. A stop event with no valid report records an unresolved validation event; the hook never invents completion, blockage, or input-request meaning from process state.
- Claude submits a typed `input_request` through the same experimental helper when it needs clarification. The request is durably persisted before optional notification and leaves the interactive client active. It routes first to the originating Amp coordinator, but automatic tmux response injection is unavailable by default because current supported Claude hooks do not prove an empty, focused, atomically reserved composer. A correlated answer counts against the follow-up budget and remains pending for manual interaction unless a later detected vendor capability can establish exact safe-input semantics.
- Each experimental receipt retains only the launch metadata needed to verify and recover its own Claude client: Claude session identity when available, exact tmux resource identity, workdir, launch-command digest, detected capabilities, and originating Amp thread. This is receipt-scoped state, not a Claude registry or an extension of worker configuration; repeated recovery friction must provide evidence before persistent client lifecycle is designed.
- A Claude client may be acquired either as a newly launched local session or by pulling a Claude Web session into a local interactive client with `claude --teleport <session-id>`. Teleport identity and provenance are retained in the receipt, but amux manages only the resulting local tmux client and does not claim lifecycle ownership of the earlier vendor-hosted execution. Because teleport verifies the repository, fetches and checks out the cloud branch, and may otherwise prompt to stash dirty changes, `/amux` requires a dedicated clean worktree, the same authenticated Claude account, and an available pushed branch; it fails closed rather than accepting an interactive stash or altering an occupied workdir.
- After the local read-only and mutating checkpoints, optional teleport experiments support two explicit acquisition paths that converge on the same receipt-scoped local model. Normally `/amux` receives the cloud session ID, takes exclusive ownership of a clean dedicated repository/worktree/branch tuple, and launches `claude --teleport`; alternatively it may adopt an already teleported interactive pane only after verifying repository, workdir, branch ownership, Claude process and session provenance, exact tmux identity, and exclusive edit ownership. Adoption is never inferred merely from finding a pane whose current command is `claude`.
- Routing is explicit at teleport acquisition. A teleport launched from an Amp thread defaults to that thread as coordinator; adoption of independently started Claude Web work requires selecting an Amp coordinator or deliberately choosing the machine-local inbox. `/amux` may later reassign the target while preserving provenance, but never infers one from repository, work-group membership, active window, or latest sender.
- After teleport completes, `/amux` captures an acquisition baseline: resulting branch and HEAD, worktree status, remote tracking state, and any known review or pull-request reference. The coordinator then explicitly declares the expected handoff for the local continuation. Subsequent reports distinguish artifacts already present in the cloud handoff from changes made after local acquisition instead of attributing the entire branch to the managed continuation.
- A managed teleport remains in pending acquisition until a supported hook or otherwise verifiable state confirms that the local interactive session is ready. Authentication, repository confirmation, Git preconditions, and other vendor prompts produce factual acquisition-needs-input attention for the selected coordinator or local inbox; `/amux` neither scrapes output to guess a response nor closes the window. Only after explicit resolution does it capture the baseline and enable managed report or parking actions.
- Local park or removal acts only on the verified local tmux client and its receipt-scoped state. The cloud session ID remains as provenance, but amux never archives, deletes, or claims to synchronize the vendor-hosted session. Any future Anthropic operation on that remote session requires a supported explicit vendor API and a separate user request rather than being implied by local lifecycle.
- During the pilot, report persistence or notification does not immediately park Claude. The interactive client remains available until the originating Amp coordinator acknowledges the report; `/amux` then verifies the receipt-scoped tmux identity and explicitly parks it while preserving the Claude session ID for possible follow-up. Experiments measure this acknowledgement-to-park step before the long-term automatic-parking policy is implemented.
- Capacity policy is staged with the pilots. The first read-only delegation may proceed after CodexBar's available windows and confidence are captured, shown, and explicitly acknowledged even though reserve floors are not yet tuned. Before the mutating pilot, provisional hard floors for five-hour, weekly, and applicable model-specific windows must be configured; each decision records the tightest governing window so later policy changes are based on observed subscription use.
- Experimental receipt retention is state-aware. Unresolved or still-recoverable receipts do not expire; after acknowledgement and parking, raw records remain for a configurable evidence window initially set to 30 days, then become eligible for deterministic cleanup. Curated aggregate findings and selected artifact or thread references may outlive raw private records without preserving prompts, transcripts, or tool streams.
- amux does not launch or supervise Tycho-style headless task runs. Future agent work stays in full interactive vendor clients so it retains their capabilities and subscription-backed usage. This does not alter ADR 0001 runners: a runner provisions a machine and workdir for vendor-owned remote execution rather than becoming an amux-dispatched headless task client.
- tmux is permanent substrate — as fixed a dependency as Git — and, with local configuration, remains the source of truth. Future designs may lean on any tmux primitive without abstraction guilt. The durable moat is restated as tmux-native substrate with a multi-surface interface: the terminal stops being the only surface, never the source of truth.
- The normal operator workflow is bootstrap through `amux launch`, then coordination from Amp's Web UI through the `/amux` skill. Tmux windows are primarily durable execution and inter-agent communication substrate, not the maintainer's main interaction surface; direct tmux navigation and indicators are fallback and diagnostic affordances.
- The definition of done is that client lifecycle becomes invisible. After the accepted lifecycle plan lands, new work is justified by felt friction in real use, not by a roadmap.

## Boundary constraints on future ideas

- Dispatch is the vendor's job. Amp or Agents Anywhere routes remote work; amux provisions runners that make the right machines and workdirs available. The runner boundary in ADR 0001 is philosophical, not temporary.
- The versioned `--json` envelope from ADR 0001 is the platform surface for `/amux`, adapters, and agent-driven coordination. No separate amux remote-control application is planned.
- GitHub awareness stays in the skill layer. The CLI core never learns about pull requests, CI, or merges.
- Final merge authority is per-project skill policy, provisionable through private `fleet` configuration where appropriate. The conservative default requires explicit human approval; selected repositories or branches may explicitly permit automatic merge only after their declared review, CI, and protection requirements pass. This policy does not enter the amux CLI core.
- A finish-to-merge checkpoint is a minimal resumable pointer, not a second workflow database: change or review identity where applicable, last attempted phase, stopping reason, and next condition. On resumption, the skill reconstructs current truth from the project's declared authoritative surfaces rather than trusting cached status. GitHub is one possible surface, not a universal assumption; a project may also use local or symlinked Markdown notes from a private repository. Checkpoints preserve references and provenance without copying private note contents into public repositories, forge metadata, or unrelated Amp threads.
- The coordinating Amp thread is the default checkpoint destination. A project may additionally declare a suitable durable sink, such as a private Markdown note or forge surface, when recovery must span threads or machines. `/amux` writes the same bounded checkpoint under one correlation identity to both destinations, reports partial delivery, and never treats either copy as authoritative CI, review, or approval status.
- Authoritative project sources are declared in layers rather than discovered by scanning the filesystem. Versioned repository instructions name portable, project-visible sources; private `fleet` configuration may overlay confidential or machine-specific sources such as symlinked operational notes. `/amux` resolves these declarations, reports every source and its provenance in diagnostics, and reads no undeclared neighboring notes.
- Layering is monotonic for safety and approval policy: a private source may add operational context or tighten a project-visible constraint, but it cannot silently weaken one. `/amux` combines compatible declarations with provenance; an irreconcilable contradiction blocks the affected action and returns the conflicting sources to the coordinating Amp thread rather than choosing a generic last-writer-wins precedence.
- Reusable instructions compose through the same explicit ownership layers: amux skills provide generic workflow fragments, repositories declare project conventions, and private `fleet` overlays add personal or machine-specific context. Each spawn or delegation resolves a deterministic, inspectable instruction bundle with source provenance; no layer is discovered implicitly and no generated bundle changes worker identity.
- Instruction composition uses typed resolution rather than raw concatenation or global last-writer-wins precedence. A project may explicitly replace a generic workflow preference, and a private overlay may specialize personal or machine context within its declared scope; safety, privacy, and approval constraints remain monotonic and can only tighten. Any unclassified or irreconcilable conflict blocks the spawn or delegation and exposes both sources to the coordinator.
- Resolved bundles are selected deterministically under an explicit context budget using declared task, backend, and workflow applicability. Mandatory safety, privacy, and approval fragments are always included and never silently truncated; omitted optional fragments remain visible in the resolution report. If mandatory content exceeds the budget, `/amux` blocks for an explicit decision instead of spending another model turn to summarize instructions or asking the worker to infer missing constraints.
- Each spawn records the identities or digest of its resolved instruction bundle. An active worker keeps that snapshot when source fragments later change; `/amux` may show a provenance-aware diff and send an explicit correlated refresh when a useful convention or urgent safety correction should apply mid-thread. Updates are never injected silently, and future restarts or follow-ups do not erase which instructions governed earlier work.
- Instruction fragments share semantic project constraints and task intent where possible while allowing explicit backend variants for Amp- or Claude-specific lifecycle, reporting, delivery, and tool behavior. Resolution selects a declared variant deterministically from detected capabilities; it neither forces false backend parity nor spends a model turn translating generic instructions.
- The fragment schema and deterministic resolver are versioned with the `/amux` skills rather than absorbed into lifecycle CLI core or delegated to model judgment. Structured dry runs and diagnostics expose selected variants, provenance, overrides, omissions, conflicts, and budget use; the skill supplies workflow intent while the helper performs mechanical resolution.
- The no-resident-supervisor principle holds: attention signals are event-driven through tmux or agent-CLI hooks, with no resident amux process. Amp Web and vendor-owned runners remain the cross-machine surface; amux makes `/amux` reliable there instead of competing with another UI or service.
- Vendor-native plugins or hooks are authoritative for semantic client events; tmux supplies backend-neutral structural fallback signals such as pane exit or activity. amux normalizes both with source provenance but never derives task meaning from tmux activity alone or takes ownership of agent execution as Tycho does.
- Attention state separates runtime observation from workflow meaning. amux may report factual client states such as idle or waiting; only an explicit agent or workflow-level signal may claim that the work itself finished.
- Attention signals remain unresolved until their source emits a superseding state. Delivery alone does not mark a signal seen; the transition occurs when the target Amp agent begins processing the correlated message, `/amux` consumes it from the local inbox, or direct tmux focus observes it as a fallback. Seen never pretends that the underlying need was resolved.
- Anomaly detection is layered and preserves confidence and provenance. Initial prototypes use only explainable structural signals such as an unexpected client exit, failed hook, lost pane, explicit agent failure, or workflow-defined deadline; semantic output interpretation and generic time thresholds remain optional later experiments.
- Attention covers interactive worker task states and runner operational faults as distinct categories. A runner signal never implies knowledge of the remote agent threads it enables; their task attention remains vendor-owned without a supported vendor event contract.
- amux does not own global tmux presentation. Built-in tmux activity or bell signals provide a zero-configuration baseline, while stable amux-published session or window metadata lets optional `fleet`-managed tmux configuration render richer attention states.
- amux ships and versions its vendor attention adapters and manages them only through explicit, dry-runnable install, update, and diagnostic commands. The accompanying `/amux` skill provides the agent-first workflow around those deterministic commands; `fleet` may invoke the same flow across machines without reimplementing the adapters.
- Claude capacity reserve floors are deterministic amux machine configuration with conservative defaults, visible to diagnostics, dry runs, and structured delegation decisions. `/amux` may inspect and update them, and `fleet` may provision the chosen values without owning the policy.
- Each machine reports a runtime capability matrix rather than inheriting a platform preset: interactive lifecycle, semantic hooks, safe tmux delivery, trustworthy capacity data, report validation, and automatic parking are independently detectable. Diagnostics, dry runs, structured output, and `/amux` explain which capability enabled or blocked an action.
- Attention persists the current materialized state per resource, including whether it has been seen, plus a bounded local event history for provenance and adapter diagnosis. It is not a permanent activity or audit archive.
- An ordinary Amp worker routes attention to its originating spawner thread when that optional provenance is known. `/amux` may explicitly reassign this routing target without changing worker identity or remote thread ownership. A worker without a usable origin, or whose configured target rejects delivery, leaves the unresolved signal in the machine-local attention inbox with the intended target and failure reason for deliberate recovery; no workspace manager persona, latest-sender inference, peer broadcast, automatic substitution, or indefinite retry loop is introduced.
- The coordinating Amp agent, operating through `/amux`, is the primary attention consumer. It handles routine coordination and routing; requests explicitly requiring human judgment and unresolved high-confidence anomalies escalate to the maintainer in Amp's Web UI. Peer workers do not receive an indiscriminate attention broadcast.
- amux's coordination ceiling is a small deterministic vocabulary for inspecting attention, addressing a worker or thread, sending a follow-up, spawning delegated work, and invoking existing lifecycle operations. Planning and judgment remain in the coordinating Amp thread; amux does not add assignments, dependency graphs, claim/complete semantics, formulas, role hierarchies, or supervisor loops.
- A Claude delegation carries a minimal, bounded delivery receipt: originating Amp thread, target Claude client or session, correlation identity, current delivery status, and eventual report reference. This state exists only to make report-back reliable across restarts; it expires after completion and does not become a durable task record or work graph.
- When an originating Amp thread already belongs to a #118 work group, a delegation receipt or attention event may reference that group for correlated inspection and routing. Group membership is optional: ungrouped work remains fully valid, and amux never creates a synthetic group merely to support one delegation or signal. The work group remains an association of canonical Amp threads rather than becoming the delegation's identity or a mandatory task container.
- #118 work-group reports, Claude delegation reports, and attention events may share only a small delivery envelope: correlation identity, producer and target, timestamp, delivery or acknowledgement state, provenance, and a typed payload. Their workflow status vocabularies remain independent—`ready`, `blocked`, and `merged` do not become universal amux states. `/amux` may derive a coarse attention category for presentation while preserving the source payload and provenance rather than rewriting its meaning.
- During experimentation, the receipt also retains bounded decision evidence: task type, declared handoff, capacity source and before/after values, coordination-turn count, human escalations, delivery and recovery outcomes, and whether the result proved useful. Full prompts, transcripts, and tool streams are excluded.
- Raw experiment records remain bounded local state. A future promotion proposal may commit only a curated aggregate summary and representative thread or artifact references, never the full operational record set.
- Report-back persists to the receipt before delivery, then uses the supported Amp cross-thread append surface when available. A verified active local worker may also receive feedback through its tmux window, and `/amux` can recover any undelivered report. Every transport preserves the receipt's correlation identity so retries are duplicate-safe.
- Automatic tmux delivery fails closed. It requires exact resource identity, the expected interactive client and workdir, and an adapter-reported safe input state; an ambiguous target receives nothing and leaves the receipt pending for `/amux` recovery.
- Reports are concise and structured around outcome, summary, blockers or questions, changed artifacts, verification, and references. Full transcripts, logs, and artifact contents stay out of the originating Amp thread unless requested, limiting coordination token overhead while preserving drill-down paths.
- After a completed report is durably recorded and delivered, the Claude Code client parks automatically while preserving its session and configuration for follow-up. A client waiting for input remains active and unresolved rather than being mistaken for completed work; explicit removal is a later coordinator decision.
- A Claude Code input request returns first to the originating Amp coordinator, which may answer from existing delegation context. Missing information, product judgment, and other decisions the coordinator cannot safely derive escalate to the maintainer in Amp's Web UI rather than being guessed.
- Each delegation sets a small maximum for automatic Amp-to-Claude follow-up turns, with a conservative default of one. Budget exhaustion leaves the delegation unresolved and escalates instead of silently consuming more tokens; the counter remains receipt metadata, not a workflow engine.
- Workdir isolation follows capability: a provably read-only Claude delegation may share the originating Amp workdir, while any delegation allowed to mutate files requires a dedicated worktree or an exclusive handoff proving Amp will not edit concurrently. Ambiguous write ownership fails closed.
- Git and worktree creation, integration, and cleanup remain `/amux` skill orchestration, following the accepted sprawl and finish boundary. The CLI validates the prepared workdir and owns the interactive client and bounded receipt, but does not absorb Git workflow judgment.
- Every mutating delegation declares its expected handoff shape in advance: clean commit, intentionally dirty exploratory worktree, or patch. A clean verified commit is the default; completion requires the declared shape rather than accepting whatever state happens to remain. This is being explored in an [ongoing Amp thread](https://ampcode.com/threads/T-019f6131-19be-7519-823d-cf0a67c33ba9), linked without treating its contents or outcome as an accepted decision.
- Structured reporting uses layered validation. Claude writes the semantic report; its adapter hook checks required fields and objective invariants such as the declared handoff shape. Invalid output keeps the receipt unresolved for Amp interpretation instead of triggering a separate normalization turn or being mislabeled complete.
- A completed delegation guarantees a valid report was delivered and its declared handoff shape and objective invariants were satisfied. It does not claim the work is correct, accepted, or ready to merge; those judgments remain with the originating Amp workflow, tests, review, and CI.

## Post-plan horizons, in priority order

1. **Amp-to-Claude delegation — prove one concrete coordination loop.** Begin with the separate [interactive Claude Code delegation pilot](../proposals/claude-code-interactive-delegation-pilot.md) and compare its evidence with #118's delivered report mechanics; reuse compatible invariants without broadening its Amp-specific workflow API, and do not add a first-class Claude resource first. One originating Amp thread may capacity-consciously delegate one task to one full interactive Claude Code client, receive bounded questions, and get a validated concise report back through a durable receipt before the Claude client parks. This slice adds only the attention states required by that loop and keeps `/amux` as the primary interface. Promotion gate: repeated real-use experiments must cover both read-only and mutating delegations, routine question routing, validated handoffs, duplicate-safe report delivery, automatic parking, and recovery from at least one real delivery or lifecycle failure while demonstrating lower coordination effort and bounded token overhead. Promotion still requires its own post-ADR-0001 decision.
2. **Generalized attention routing — generalize only from delegation evidence.** Extend the proven event and receipt model to answer which workers need input, finished a handoff, or encountered a high-confidence anomaly, and which routine actions an Amp coordinator can take. Human escalation remains tiered; native tmux indicators and a human-readable aggregate remain fallback and diagnostic surfaces, not the primary control plane. Promotion gate: evidence that the delegation-specific states generalize cleanly across Amp workers and runner operational faults without adding a daemon, abstract work graph, or prompt-heavy supervisor.
3. **Finish-to-merge pipeline.** One on-demand `/amux` skill workflow shepherds the tail of a worker through pull-request creation, review feedback, CI repair, merge readiness, merge under the chosen approval policy, and then the already accepted finish cleanup. It polls only for a bounded interval while an invocation remains useful; longer CI or review waits produce a concise durable checkpoint and stop, so a later `/amux` invocation can resume without reconstructing the workflow. A future supported event integration may trigger that resumption, but correctness never depends on one. The workflow introduces no resident watcher, scheduler, or GitHub concepts in the CLI core. Promotion gate: repeated use must show that resumable skill state reduces tail-work coordination without making GitHub or workflow state an amux lifecycle resource.
4. **Instruction consistency and named recipes.** Generalize from sprawl instruction templates into reusable instruction fragments, per-project defaults for backend and conventions, and optionally named launch recipes that select those inputs. A recipe configures a newly spawned worker but has no memory, routing, role ownership, or identity of its own; every resulting Amp worker remains a distinct canonical thread. The special-agent workflow in [#118](https://github.com/zainfathoni/amux/issues/118) is early evidence for this direction: it encodes reusable skill orchestration atop distinct work-group, coordinator-thread, and worker-thread identities rather than creating a persona. Promotion gate: repeated use must show that consistent instructions and recipes reduce spawning and delegation correction turns without hiding resolved provenance, moving planning judgment into CLI core, or conflating a recipe with a work group or worker.

## Open tensions

- Named recipes are configuration, not agents. A durable persona with memory, routed ownership, or identity spanning threads would conflict with the accepted contract that a worker's identity is its canonical Amp thread ID; no such persona is planned without new evidence and a separate identity decision.
- A prompt library and per-project defaults strain the stance that instructions stay in the skill layer and configuration stays instruction-free. The configuration surface may need to grow, which is a trade-off a future ADR must own.

## Consequences

Recording this direction constrains future proposals without committing any of them: ideas that violate the boundary constraints are deferred rather than silently adopted, and each horizon carries an explicit promotion gate. The JSON contract gains weight as the stable seam between deterministic local lifecycle and agent-driven coordination, not as a promise of a separate control application. Nothing in this document changes accepted interface rules, in-flight work, or the domain glossary; future-facing vocabulary used here (interactive client, Claude Code client, named durable agent) enters CONTEXT.md only if and when its horizon is promoted.
