The /amux skill

The bundled skill gives agents a precise vocabulary for local Amp lifecycle. It separates configuration, local execution, and remote thread state—and reserves higher-level health, sprawl, and finish behavior for safe skill orchestration.

Install globally

npx skills add zainfathoni/amux --skill amux --global

skills.sh is the primary installation path. Reload Amp if the new skill is not indexed immediately. Local symlinking is only a contributor workflow documented in the repository’s CONTRIBUTING guide.

Terminology

  • Worker: interactive TUI client identified by canonical Amp thread ID.
  • Runner: non-interactive amp --no-tui client identified by canonical workdir. It enables remote work but does not own remote agent threads.
  • Workspace: worker/runner lifecycle group represented by one same-named tmux session.

Top-level list, launch, park, restart, remove, doctor, and reconcile aggregate both modes. The worker and runner namespaces narrow scope. Pin/unpin always require a mode namespace.

Skill-only health

/amux health aggregates workers and runners by default, with optional workspace and mode filters. It is not a CLI command.

  • Workers: match configuration, tmux placement, workdir, thread, and an idle interactive Amp pane; send one tokenized read-only prompt only when safe.
  • Runners: never send input; verify canonical workdir, ownership, generated window, and the exact amp --no-tui process.

A worker timeout means candidate stale, not safe to replace. Health never mutates, launches, restarts, removes, reconciles, or spawns.

Skill-only sprawl

/amux sprawl remains worker-only. Before side effects, the agent reads every issue and native dependency relationship, checks likely file/API overlap, and sequences anything dependent or conflicting.

amux spawn --workspace <workspace> \
  --window install-diagnostics \
  --workdir <dedicated-worktree> \
  --mode medium \
  --title-prefix '#110' \
  --group amux-110 \
  --message-file <assignment> \
  --idempotency-key issue-110

Every accepted issue gets the amux-agent-first label, one narrow assignment, dedicated branch/worktree, semantic issue-unprefixed window, focused Oracle diff review, and explicit callback/finish protocol. For Amux issue work, the durable group/Amp label is amux-<issue-number> and each report ID is amux-<issue-number>-worker-<ordinal>. Other unconfigured repositories use an explicit equivalent lowercase, group-safe repository slug. A repository may instead configure tracker-neutral <project-prefix>-<work-item-id>-<short-slug> naming in group-naming.json; --work-item-id and --worker-ordinal resolve the group and exact <group-id>-worker-<ordinal> report before mutation. Scope comes only from the workdir’s verified origin, never cwd/worktree/tmux naming or prior labels. Explicit --group remains authoritative. This workflow convention does not restrict generic group IDs or migrate existing amux-*, repository-slug, legacy issue-*, purpose-specific, or explicit groups. Medium is mandatory unless the user explicitly requests another mode; complexity never raises it automatically.

Durable issue-worker coordination

The coordinator first inspects native issue dependencies and active overlap, preflights configured naming when requested, then creates worktrees from fresh origin/main. It declares a durable group without projecting the long-lived coordinator to an Amp label, registers the exact verified coordinator pane, and spawns with automatic naming or explicit --group so only the authoritative receiving thread is attached and labelled.

amux callback register --group amux-135 --thread <coordinator> --pane <pane>
amux spawn --mode medium --workspace <workspace> --window <semantic-window> \
  --workdir <fresh-worktree> --title-prefix '#135' \
  --group amux-135 --message-file <assignment> --idempotency-key issue-135
amux report submit --report-id amux-135-worker-1 --group amux-135 \
  --thread <worker> --status ready --issue '#135' --pr <url> \
  --summary implementation-tests-review-pr-ci-complete

A report is persisted before the short wake-up token. Notification and acknowledgement are not finish authorization. The coordinator independently verifies PR head/scope/mergeability/closing issue, worktree and CI, then verifies post-merge CI before recording explicit durable authorization. The child progresses the same report to terminal merged; /amux finish performs safe Git/worktree cleanup and teardown last. Group/report history survives.

Exit 2 lock contention writes nothing: wait, then retry the identical operation and report ID. Missing, stale, recycled, duplicate, reordered, and restarted callbacks fail closed or recover from durable state. Never retry into a suspected busy composer; inspect pending/history directly. Never guess a pane, force-delete, auto-release, or infer finish from a wake-up.

Soft budgets to ready are Small 30m, Medium 1h, and Large 2h; split XL. Stale is 15m, review warns after 10m, demonstrated external CI alerts after 20m, and authorized finish alerts after 10m. Only demonstrated external waits pause active time. One extension may add at most half the original budget under a new generation; expiry is diagnostic only. The coordinator owns one nearest-deadline queue, not a timer process per child. This is policy, not a CLI deadline option; never edit reports.json directly.

Skill-only finish

/amux finish verifies merge and a clean worker worktree, then fails closed if runner configuration or diagnostics show that the worktree has acquired runner ownership. It never removes or unpins that runner implicitly.

After updating main, finish removes the clean worktree without force and preserves squash-merge branch safety: git branch -d first; no automatic -D. Worker teardown runs last. A ready callback never authorizes cleanup; finish requires independently verified merge and explicit authorization.

Agent contracts

  • Use long canonical selectors and current contextual help.
  • Use --dry-run before mutation and --json for parsing.
  • JSON v1 separates planned, successful, skipped, and failed.
  • Exit 2 rejects before mutation; exit 1 may represent partial runtime success.
  • Mutations are idempotent desired state under one machine lock.
  • Spawn requires a stable idempotency key; reports require a stable report ID.
  • Group labels are add-only and member-only; coordinator roles stay local, while local removal or later promotion may leave permanent external drift.
  • Health, coordinator orchestration, sprawl, and finish are not amux subcommands.
